# 2026-04-21 — Plan 22: `bonsai init` cinematic redesign — Phases 1 + 2

## Context

Plan 22 is a 5-phase cinematic rewrite of `bonsai init`: 4 staged forms (Vessel/Soil/Branches/Observe), Japanese kanji/kana stage labels, Enso-rail progress, live file-tree preview. All behind `BONSAI_REDESIGN=1` until Phase 5 flips default. User asked to review + refine the plan, then ship Phase 1 via `/issue-to-implementation`. After `/context` check authorized Phase 2 in the same session.

## Decisions

- **Pre-dispatch plan commit.** Agent worktrees base off `origin/main`, not local main (memory learning from Plan 20 + Plan 21). Plan doc pushed as `525ee11` style commit before each dispatch so the agent's worktree could see it. Both phases worked first try.
- **Plan review refined 7 items before dispatch.** Locked `RenderFileTree` as the widget name (not `FileTree2`), demo test uses `t.Log` in a normal test function (not `go:build ignore`, which doesn't work for runnable demos), `strings` import added to fallback.go example, `agentDef.DisplayName` + `startedAt` threaded through stage ctors (needed for Observe stage later), legacy test sweep made explicit, `BranchesStage.Reset` semantics documented.
- **`StubStage` hygienic addition.** Agent added a `StubStage` type not in the plan's file list. Hygienic — it lets Phase 2 ship a runnable 4-stage skeleton without fabricating form logic that belongs in Phase 3/4. Kept.
- **`Chromeless` capability, not a forced interface.** Harness detects `Chromeless() bool` via type assertion. Lets legacy harness users stay unchanged; initflow stages opt in. Preserves the harness contract for existing callers.
- **ASCII fallback via `WideCharSafe()`.** Env `BONSAI_ASCII_ONLY=1` force-opt-out, runewidth EastAsianWidth probe, WT_SESSION Windows Terminal, allow-listed TERM prefixes (xterm/screen/tmux/alacritty/kitty/wezterm/ghostty), LANG UTF-8 fallback. Conservative by default — better to degrade gracefully on one terminal than render mojibake everywhere.
