## How Triggers Work

Bonsai abilities (skills, workflows, protocols, sensors) don't just sit in files waiting to be read manually. They activate through a layered system that ranges from fully automatic enforcement to prompt-based invocation. The goal: the right ability loads at the right time, without the user needing to remember file paths or instruction sets.

Each layer operates independently. A single ability can be activated through multiple layers simultaneously â€” a skill might auto-load when you open a matching file (Layer 2), appear in a navigation table with scenario descriptions (Layer 4), and contain embedded trigger hints in its own file (Layer 5).

---

## The 5-Layer Activation Stack

```
  Layer 1 â”‚ Sensors (hooks)              â”‚ Always active â€” auto-enforced
  Layer 2 â”‚ Path-scoped rules            â”‚ Auto-load when reading matching files
  Layer 3 â”‚ Claude Code skills           â”‚ /slash-command and description-based
  Layer 4 â”‚ CLAUDE.md navigation tables  â”‚ Scenario-based "Activate when..." guidance
  Layer 5 â”‚ Trigger sections in files    â”‚ Embedded scenarios, paths, and examples
```

### Layer 1: Sensors (Hooks)

Sensors are the most deterministic layer. They run automatically via Claude Code's hook system â€” no user action required.

**How they work:** Bonsai generates hook entries in `.claude/settings.json` during `bonsai init` and `bonsai add`. Each sensor script runs at a specific event (session start, before/after tool use, on stop) and can inject context, block actions, or flag issues.

**Events:**
| Event | When it fires |
|:------|:-------------|
| `SessionStart` | Once at session startup |
| `UserPromptSubmit` | Before each user prompt is processed |
| `PreToolUse` | Before the agent uses a tool (can block) |
| `PostToolUse` | After a tool completes |
| `SubagentStop` | When a dispatched sub-agent finishes |
| `Stop` | After every agent response |

**Examples:**
- `session-context` fires at `SessionStart` â€” injects identity, memory, and protocols so the agent always starts with full context
- `scope-guard-files` fires at `PreToolUse (Edit|Write)` â€” blocks the agent from editing files outside its workspace
- `context-guard` fires at `UserPromptSubmit` â€” detects session wrap-up phrases and injects behavioral constraints

Sensors are always active. You cannot trigger them manually â€” they enforce automatically.

### Layer 2: Path-Scoped Rules

Path-scoped rules auto-load skills when the agent reads files matching specific glob patterns. They live in `.claude/rules/skill-{name}.md` and are generated by Bonsai.

**How they work:** Each rule file contains a YAML frontmatter with `paths:` globs. When Claude reads a file matching any of those globs, the rule fires and instructs the agent to load the corresponding skill.

**Example:** When the agent reads a file in `**/routes/**`, the `skill-api-design-standards.md` rule fires and loads the API Design Standards skill.

This layer is automatic but only activates when the agent interacts with matching files. If the agent never reads a Go file, `skill-coding-standards.md` will never fire. See [Path-Based Auto-Loading](#path-based-auto-loading) for the complete glob reference.

### Layer 3: Claude Code Skills

Claude Code skills (`.claude/skills/{name}/SKILL.md`) enable two activation methods:

1. **Slash commands** â€” type `/{name}` to explicitly invoke a workflow
2. **Description-based auto-invocation** â€” Claude reads skill descriptions and may automatically load the workflow when the user's request matches

**How they work:** Bonsai generates `SKILL.md` files for curated workflows during `bonsai init`. Each file has a `name`, `description`, and `when_to_use` field that Claude uses for matching.

Only the [7 curated workflows](#slash-commands) get Claude Code skill files. Other workflows and all skills are activated through other layers.

### Layer 4: CLAUDE.md Navigation Tables

The generated `CLAUDE.md` in each workspace contains scenario-based navigation tables. Instead of listing abilities by name, tables use "Activate when..." descriptions derived from trigger metadata.

**Example:**

```
| Activate when... | Read this |
|------------------|-----------|
| Starting end-to-end planning for a new feature or task | agent/Workflows/planning.md |
```

This layer relies on the agent reading CLAUDE.md (which happens at session start via the `session-context` sensor) and using the scenario descriptions to decide which ability to load for the current task.

### Layer 5: Trigger Sections in Ability Files

Each skill and workflow file begins with a `## Triggers` section containing activation metadata:

- **Slash command** (workflows only) â€” the `/command` name if available
- **Auto-loads when reading** â€” glob patterns that trigger path-scoped rules
- **Activate when** â€” scenario descriptions matching user intent
- **Examples** â€” sample prompts and the expected agent action

This layer serves as in-file documentation. When an agent loads an ability file, the trigger section reinforces when and why it should be used, and provides examples of correct activation.

## Slash Commands

Seven curated workflows are available as `/slash-commands` in Claude Code. Type the command to invoke the workflow directly.

| Command | Workflow | What it does |
|:--------|:---------|:-------------|
| `/planning` | Planning | End-to-end planning process â€” from request to dispatch-ready plan |
| `/code-review` | Code Review | Review agent output against the plan â€” correctness, standards, security |
| `/pr-review` | PR Review | Review a pull request â€” context, scope, correctness, security, performance, standards |
| `/security-audit` | Security Audit | Security audit â€” secrets scan, dependency audit, SAST, config review, access control |
| `/issue-to-implementation` | Issue to Implementation | End-to-end autonomous workflow â€” issue intake through to shipped code |
| `/test-plan` | Test Plan | Design a structured test plan for a feature â€” scope, prioritize, allocate test types |
| `/plan-execution` | Plan Execution | Execute an assigned plan step by step â€” implement, test, report |

Slash commands are the most explicit activation method. Use them when you know exactly which workflow you want, or when description-based auto-invocation doesn't match your intent.

## Path-Based Auto-Loading

Skills with path triggers auto-load when the agent reads files matching their glob patterns. These are the 11 skills with path-based rules:

| Skill | Glob Patterns |
|:------|:-------------|
| API Design Standards | `**/routes/**`, `**/handlers/**`, `**/controllers/**`, `**/api/**` |
| Auth Patterns | `**/auth/**`, `**/middleware/auth*`, `**/login*`, `**/session*` |
| CLI Conventions | `cmd/**`, `**/cli/**`, `**/*command*` |
| Coding Standards | `*.go`, `*.py`, `*.js`, `*.ts`, `*.java`, `*.rs` |
| Container Standards | `**/Dockerfile*`, `**/docker-compose*`, `**/k8s/**` |
| Database Conventions | `*.sql`, `migrations/**`, `**/schema.*`, `**/models/**` |
| Design Guide | `**/components/**`, `**/*.css`, `**/*.scss`, `**/styles/**` |
| IaC Conventions | `**/*.tf`, `**/*.tfvars`, `**/cloudformation/**` |
| Mobile Patterns | `**/ios/**`, `**/android/**`, `**/*.swift`, `**/*.kt` |
| Test Strategy | `**/*_test.go`, `**/*.test.*`, `**/*.spec.*` |
| Testing | `**/*_test.go`, `**/*.test.*`, `**/*.spec.*`, `**/test/**` |

**Note:** Test Strategy and Testing share some glob patterns. Both will auto-load when the agent reads test files. Test Strategy covers the architectural decisions (which tests to write and why), while Testing covers the implementation conventions (frameworks, patterns, coverage requirements).

The remaining 6 skills (Dispatch, Issue Classification, Planning Template, PR Creation, Review Checklist, Workspace Guide) do not have path-based rules. They activate through navigation tables (Layer 4) and scenario matching (Layer 5).

## Customization

### Adding Custom Rules

You can create your own `.claude/rules/` files alongside the ones Bonsai generates. Custom rule files:

- Will not be tracked by the Bonsai lock file (`.bonsai-lock.yaml`)
- Will not be overwritten or removed by `bonsai update`
- Follow the same format as generated rules â€” YAML frontmatter with `paths:` globs, followed by instructions

**Example custom rule** (`.claude/rules/custom-logging.md`):

```markdown
---
paths:
  - "**/logging/**"
  - "**/logger.*"
---

When working with logging files, follow these conventions:
- Use structured logging (key-value pairs, not string interpolation)
- Always include a correlation ID
- Log at appropriate levels: DEBUG for flow, INFO for events, ERROR for failures
```

### Adding Custom Skills

You can create your own `.claude/skills/` files to add custom slash commands:

**Example custom skill** (`.claude/skills/deploy/SKILL.md`):

```markdown
---
name: deploy
description: Run the deployment workflow for staging or production
when_to_use: When the user wants to deploy the application
---

Follow the deployment checklist:
1. Verify all tests pass
2. Check the target environment
3. Run the deployment script
```

This creates a `/deploy` slash command that Claude Code can invoke.

### What Bonsai Manages vs. What You Own

| File pattern | Managed by Bonsai | Safe to customize |
|:------------|:------------------|:-----------------|
| `.claude/rules/skill-*.md` | Yes â€” generated, tracked in lock file | No â€” will be overwritten on `bonsai update` |
| `.claude/rules/custom-*.md` | No | Yes â€” Bonsai will never touch these |
| `.claude/rules/{any-other-name}.md` | No | Yes â€” Bonsai only generates `skill-` prefixed files |
| `.claude/skills/{workflow}/SKILL.md` | Yes â€” generated for curated workflows | No â€” will be overwritten on `bonsai update` |
| `.claude/skills/{custom}/SKILL.md` | No | Yes â€” Bonsai will never touch these |

### Version Control

Generated `.claude/rules/` and `.claude/skills/` files **should be committed to version control**. They are project-specific configuration derived from your Bonsai setup, not user-local state. Committing them ensures:

- All team members get the same trigger behavior
- CI/CD environments have consistent agent configuration
- Changes to triggers are tracked and reviewable in PRs

## Troubleshooting

### Ability didn't activate

**Symptom:** You expected a skill or workflow to load, but the agent didn't use it.

**Check these:**

1. **Is the ability installed?** Run `bonsai list` to see what's installed for each agent. If the ability isn't listed, add it with `bonsai add`.

2. **Is the agent type compatible?** Each ability has an `agents:` field in its `meta.yaml`. A backend agent can't install tech-lead-only skills like Dispatch or Planning Template. Run `bonsai catalog` to see compatibility.

3. **Did the agent read a matching file?** Path-based rules only fire when the agent reads a file matching the glob pattern. If you're asking about API design but the agent hasn't read any files in `**/routes/**` or `**/api/**`, the rule won't fire.

4. **Is the session started properly?** The `session-context` sensor loads CLAUDE.md (which contains nav tables) at session start. If you skipped the greeting, the agent may not have its full trigger context.

5. **Try explicit invocation.** Use a slash command (`/planning`, `/code-review`, etc.) to bypass auto-detection and invoke the workflow directly.

### Wrong ability loaded

**Symptom:** The agent loaded a different ability than you expected.

**Check these:**

1. **Overlapping path patterns.** Some skills share glob patterns (e.g., Test Strategy and Testing both match `**/*_test.go`). This is by design â€” both may load simultaneously. If you want only one, ask the agent explicitly: "Follow the Testing skill, not Test Strategy."

2. **Ambiguous prompt.** If your request matches multiple scenario descriptions, the agent may choose the wrong one. Be more specific, or use a slash command.

3. **Custom rules conflicting.** If you added custom `.claude/rules/` files, check whether their path patterns overlap with generated rules.

### How to check what's active

1. **List installed abilities:** `bonsai list` shows everything installed per agent
2. **Check generated rules:** Look in `{workspace}/.claude/rules/` for `skill-*.md` files
3. **Check generated skills:** Look in `{workspace}/.claude/skills/` for workflow `SKILL.md` files
4. **Check hook configuration:** Look in `.claude/settings.json` for sensor hook entries
5. **Read the trigger section:** Open any skill or workflow file â€” the `## Triggers` section at the top shows all activation methods for that ability
