## What Bonsai Actually Does

Bonsai is a **generator**, not a runtime. It runs once (or a few times) to set up the instruction layer that Claude Code agents read when they work in your project. After that, Bonsai's job is done — the agents operate using the files it created.

What it generates:

- **Identity documents** that tell each agent who they are, what they own, and how they relate to other agents
- **Working memory files** that persist context between sessions
- **Protocols** — hard rules the agent follows every session (security, scope boundaries, startup sequence)
- **Workflows** — step-by-step procedures for tasks like planning, code review, and reporting
- **Skills** — domain knowledge (coding standards, API design, auth patterns, testing conventions)
- **Sensors** — shell scripts wired into Claude Code's hook system that automatically inject context, enforce boundaries, and catch mistakes
- **Routines** — periodic maintenance tasks with a dashboard that tracks when they last ran
- **Project scaffolding** — shared documents (status tracker, roadmap, plans directory, decision log, reports) that all agents reference

The key insight: Claude Code agents are only as good as the instructions they have access to. Bonsai gives them a structured, layered instruction set so they behave consistently, stay in scope, and coordinate with each other.

## The Mental Model

### Agents as Team Members

Think of each agent as a new hire joining your team. They need:

- A **role description** (identity) — what's their job, what's their mindset
- **Company rules** (protocols) — security policy, what they can and can't touch
- **Playbooks** (workflows) — how to plan, how to review code, how to report
- **Domain training** (skills) — coding standards, design patterns, API conventions
- **Automated guardrails** (sensors) — systems that catch them before they make mistakes
- **Maintenance duties** (routines) — periodic tasks they're responsible for

Bonsai generates all of this. When an agent starts a session, it reads its identity, checks its memory, follows its startup protocol, and gets context injected by sensors — all before you even ask it to do anything.

### The Instruction Stack

Each agent's instructions are layered. Higher layers are more concrete:

```
Layer 6: Sensors          Automated enforcement (hook scripts)
Layer 5: Routines         Periodic maintenance procedures
Layer 4: Skills           Domain knowledge and standards
Layer 3: Workflows        Step-by-step task procedures
Layer 2: Protocols        Hard rules (security, scope, startup)
Layer 1: Core             Identity + memory + self-awareness
```

**Layer 1 — Core** is loaded every session. It tells the agent who they are and what they remember from last time.

**Layer 2 — Protocols** are non-negotiable rules. The security protocol says "NEVER hardcode secrets" — there's no override. The scope-boundaries protocol says "NEVER edit files outside your workspace" — enforced by sensors.

**Layer 3 — Workflows** are procedures the agent follows for specific tasks. The planning workflow tells the tech-lead how to go from a vague request to a structured implementation plan. The plan-execution workflow tells code agents how to implement step by step.

**Layer 4 — Skills** are reference material. When writing an API endpoint, the agent checks api-design-standards. When writing tests, it checks the testing skill. These aren't instructions to follow — they're knowledge to apply.

**Layer 5 — Routines** are maintenance tasks with schedules. The dependency-audit routine runs every 7 days. The agent checks its routine dashboard at session start and flags overdue items.

**Layer 6 — Sensors** run automatically via Claude Code hooks. The agent doesn't choose to run them — they fire on events like session start, file edits, or command execution. They're the automated safety net.

### The Orchestra Model

Bonsai supports two working styles:

**Solo agent** — One agent (usually fullstack) handles everything. Simpler setup, good for smaller projects or solo developers.

**Multi-agent team** — A tech-lead plans and orchestrates, code agents execute. This is where Bonsai really shines:

```
You (the human)
 └── Tech Lead agent
      ├── Backend agent    (executes backend plans)
      ├── Frontend agent   (executes frontend plans)
      └── DevOps agent     (handles infrastructure)
```

The tech-lead **never writes application code**. It reads your request, creates a plan, dispatches it to code agents (using Claude Code's Agent tool with worktree isolation), and reviews their output when they're done. The dispatch-guard sensor enforces that every dispatch has a plan reference and uses worktree isolation. The subagent-stop-review sensor triggers a structured review checklist when a code agent finishes.

This separation matters because planning and execution require different mindsets. The tech-lead thinks about architecture, trade-offs, and cross-cutting concerns. The code agents focus on clean implementation within their domain.

## How the Pieces Fit Together

### Project Scaffolding — The Shared Layer

When you run `bonsai init`, it creates shared documents that all agents reference:

**INDEX.md** — The project snapshot. Tech stack, project phase, document registry. Every agent reads this at session start to understand the project context.

**Playbook/Status.md** — Live task tracker. What's in progress, what's pending, what's done. Agents check this to avoid conflicting work and to understand current priorities.

**Playbook/Roadmap.md** — Long-term milestones and phases. Helps agents understand where their current task fits in the bigger picture.

**Playbook/Plans/Active/** — Where implementation plans live. The tech-lead writes plans here; code agents read them. Plans are numbered (001, 002...) and contain step-by-step implementation instructions.

**Logs/KeyDecisionLog.md** — Settled architectural decisions. When the team decides to use PostgreSQL over MongoDB, it goes here. Agents check this before proposing alternatives to settled questions.

**Logs/FieldNotes.md** — Your notes to the agents. If you did work outside of agent sessions (manual changes, meetings with stakeholders, new requirements), write it here. Agents read this at session start.

**Reports/Pending/** — Where code agents drop completion reports after finishing a plan. The tech-lead reviews these during code review.

### Agent Workspaces — The Private Layer

Each agent gets its own directory (e.g., `backend/agent/`). This is its workspace — the files it owns and is allowed to modify. The directory contains:

- **Core/** — Identity, memory, self-awareness. Loaded every session.
- **Protocols/** — Hard rules. Referenced by identity, enforced by sensors.
- **Workflows/** — Task procedures. The agent follows these when you ask it to plan, execute, review, etc.
- **Skills/** — Domain knowledge. Referenced as needed during work.
- **Sensors/** — Rendered shell scripts. Wired into `.claude/settings.json` as Claude Code hooks.
- **Routines/** — Maintenance procedure files. Tracked by the dashboard in `Core/routines.md`.

### The Config File — .bonsai.yaml

This is the single source of truth for what's installed. It tracks:

- Project name and description
- Docs path
- Selected scaffolding items
- Every installed agent with its type, workspace, and selected components

You shouldn't need to edit this by hand — `bonsai add` and `bonsai remove` manage it.

### Settings and Hooks — .claude/settings.json

Bonsai auto-generates this file with hook entries for every installed sensor. When Claude Code fires an event (session start, file edit, command run), the matching sensor scripts execute automatically. This is how sensors enforce boundaries without the agent choosing to cooperate.

## Interaction Patterns

This is the most important section. The agent system Bonsai generates is powerful, but you get the best results by knowing how to talk to your agents. These patterns will help you trigger the right behaviors and get consistent, high-quality output.

### Starting a Session

**Always open with a greeting like "Hi, get started" before giving your actual task.**

```
You:  Hi, get started
Agent: [reads identity, memory, protocols, checks status, reviews pending items...]
You:  Great. Now here's what I need...
```

Why this matters: The session-context sensor fires at session start and injects the agent's core files, protocols, project status, and health checks. But the agent needs to actually _process_ all that context before diving into work. A greeting gives it a moment to orient itself — read its memory, check what's in progress, notice any warnings.

If you skip the greeting and jump straight into a task, the agent may not thoroughly process its startup context. It might miss that there's an overdue routine, a pending report to review, or a status update that changes priorities.

**What the agent does on startup:**

1. Session-context sensor fires — injects identity, memory, protocols, project INDEX, status, field notes
2. Agent reads its Core/identity.md and Core/memory.md (instructed by the session-start protocol)
3. Checks Playbook/Status.md for current work state
4. Checks Logs/FieldNotes.md for anything you've noted
5. Reviews any warnings (stale memory, pending reports, overdue routines)

After this, the agent knows who it is, what happened last session, and what the current project state looks like. Now it's ready for your actual request.

### Planning Work (Tech Lead)

When you need something built, start with the tech-lead. Be specific about what you want but leave the _how_ to the agent.

```
You:  Hi, get started
Lead: [processes startup context]
You:  I need user authentication added to the app. Email/password login,
      JWT tokens, and a protected dashboard route.
Lead: [follows planning workflow: researches codebase, surfaces questions,
      writes a structured plan, self-reviews it]
```

**Good planning prompts:**
- "I need [feature]. Here's what it should do: [requirements]."
- "We have a bug where [symptom]. Investigate and plan a fix."
- "Refactor [area] to [goal]. Keep backward compatibility."

**What to expect:** The tech-lead will ask clarifying questions before writing the plan. Answer them — this is the most valuable part of the process. The plan ends up in `Playbook/Plans/Active/` as a numbered document with step-by-step implementation instructions.

**After the plan is written:**
- Review it yourself. The plan is a markdown file — read it, suggest changes, ask for revisions.
- Once you're happy, tell the tech-lead to dispatch it.

### Dispatching Work to Code Agents

Once a plan is ready, tell the tech-lead to dispatch:

```
You:  Dispatch plan 003 to the backend agent
Lead: [dispatch-guard validates: worktree isolation? plan reference? plan exists?]
      [dispatches backend agent with plan context in an isolated worktree]
```

The dispatch-guard sensor enforces that every dispatch:
1. Uses **worktree isolation** — the code agent works on a branch, not directly on main
2. References a specific plan
3. The plan file actually exists

This means the tech-lead can't just wing it — there must be a plan, and the work must be isolated.

**When the code agent finishes**, the subagent-stop-review sensor triggers automatically, giving the tech-lead a structured review checklist:
1. Review output against the plan — every step followed, nothing improvised
2. Check security against SecurityStandards.md
3. Verify that verification steps from the plan passed
4. Log results, update status, process any pending reports

### Executing Plans (Code Agents)

If you're working directly with a code agent (not through the tech-lead), reference the plan:

```
You:  Hi, get started
Agent: [processes startup context]
You:  Execute plan 003
Agent: [reads the plan from Playbook/Plans/Active/, follows plan-execution
       workflow: implement step by step, test, report]
```

The agent will:
1. Read the plan and understand all steps
2. Implement each step, checking off as it goes
3. Run tests
4. Write a completion report to Reports/Pending/
5. Update Status.md

**If you want partial execution:**
- "Execute steps 1-3 of plan 003"
- "Just do the database migration part of plan 003"

### Reviewing Work

After code agents finish (either dispatched or manual), review their output:

```
You:  Hi, get started
Lead: [notices pending reports in Reports/Pending/]
You:  Review the backend agent's work on plan 003
Lead: [follows code-review workflow: loads plan, checks completeness,
       reviews code quality, security, standards compliance, gives verdict]
```

The tech-lead's code-review workflow produces a structured verdict:
- **Pass** — merge the work
- **Revise** — specific changes needed (sends back to code agent)
- **Escalate** — architectural issue that needs human decision

**You can also ask for targeted reviews:**
- "Review just the API endpoints from plan 003"
- "Check if the auth implementation follows our security standards"
- "Are there any test coverage gaps?"

### Running a Security Audit

The security agent (or any agent with the security-audit workflow) can do targeted audits:

```
You:  Hi, get started
Agent: [processes startup context]
You:  Run a security audit on the authentication module
Agent: [follows security-audit workflow: secrets scan, dependency audit,
       SAST, config review, access control review]
```

**Scoped audits work better than broad ones:**
- "Audit the API routes for injection vulnerabilities"
- "Check our dependency tree for known CVEs"
- "Review the session management code against OWASP guidelines"

### Checking and Running Routines

If any routines are installed, the routine-check sensor fires at session start and flags overdue ones:

```
You:   Hi, get started
Agent: [startup context includes: "OVERDUE: dependency-audit (last ran 12 days ago, due every 7 days)"]
You:   Run the dependency audit
Agent: [follows the routine procedure, logs results to Logs/RoutineLog.md,
        updates the dashboard in Core/routines.md]
```

**You can also proactively ask:**
- "Are any routines overdue?"
- "Run all overdue routines"
- "Run the vulnerability scan routine"

Each routine has a concrete procedure — it's not freeform. The agent follows specific steps (scan dependencies, cross-reference CVEs, log findings, update dashboard).

### Checking Project Status

Your agents can give you a project overview at any time:

```
You:  What's the current project status?
Agent: [reads Status.md, Roadmap.md, checks pending reports, reviews recent logs]
```

**More specific status queries:**
- "What's in progress right now?"
- "Are there any pending reports I should review?"
- "What's next on the roadmap?"
- "Show me the last few decisions from the decision log"

### Sharing Context Between Sessions

The agent's memory resets between conversations. Three things bridge the gap:

**1. memory.md** — The agent's working memory. At the end of each session, the agent should update this with current work state, flags, and notes. At the start of the next session, it reads this to pick up where it left off.

**2. FieldNotes.md** — Your notes to the agent. If you did work outside a session, made decisions in meetings, or have new requirements, write them here. The agent reads this at every session start.

**3. Status.md** — The task tracker. Shows what's done, in progress, and pending. Updated by agents during sessions.

**If the agent seems to have lost context:**
- "Re-read your memory file"
- "Check FieldNotes — I added some notes"
- "Review the current status"

### Ending a Session Well

Before closing a conversation, prompt the agent to wrap up:

```
You:  Let's wrap up this session
Agent: [updates memory.md with current state, writes session log,
        files completion report if applicable, updates Status.md]
```

This ensures the next session starts with full context. The session-logging workflow gives the agent a structured format for recording what was done, decisions made, and open items.

**If you forget to wrap up**, the next session's startup will still work — the agent reads memory and status. But the context will be less precise because the previous session didn't explicitly record its state.

### Working with Multiple Agents

When you have a tech-lead orchestrating code agents, a typical feature cycle looks like this:

```
Session 1 — Tech Lead (planning)
  You:  Hi, get started
  You:  I need a REST API for user profiles with CRUD endpoints and pagination
  Lead: [asks clarifying questions, writes plan 004]
  You:  Looks good, dispatch to backend

Session 2 — Tech Lead (review)
  You:  Hi, get started
  Lead: [notices backend agent finished, pending report in Reports/Pending/]
  You:  Review the backend work on plan 004
  Lead: [reviews, verdict: pass with minor suggestions]
  You:  Approve it. Now plan the frontend — profile page with edit form
  Lead: [writes plan 005, dispatches to frontend]

Session 3 — Tech Lead (integration)
  You:  Hi, get started
  You:  Review frontend work on plan 005, then do a security review of both
```

**You don't have to go through the tech-lead for everything.** For quick fixes or small tasks, talking directly to a code agent is fine:

```
You:  Hi, get started
You:  Fix the pagination bug in the /users endpoint — it returns duplicates
      when sorting by created_at
Agent: [investigates, fixes, tests, reports]
```

The tech-lead shines when work needs planning, coordination, or review across agents.

## Understanding Sensors

Sensors are the automated enforcement layer. They run as Claude Code hooks — shell scripts that fire on specific events. You don't invoke them; they just work.

### How They Work

Each sensor has an **event** (when it fires) and an optional **matcher** (which tool triggers it):

| Event | When it fires |
|-------|--------------|
| `SessionStart` | Beginning of a conversation |
| `PreToolUse` | Before Claude Code executes a tool (can block it) |
| `PostToolUse` | After a tool executes |
| `SubagentStop` | When a dispatched subagent finishes |

Sensors with `PreToolUse` event can **block actions** by exiting with code 2. This is how scope-guard-files prevents editing outside the workspace — it checks the file path, and if it's out of bounds, the edit is rejected before it happens.

### Key Sensors

**session-context** (SessionStart) — The foundation. Injects the agent's identity, memory, all protocols, project INDEX, current status, field notes, and health warnings (stale memory, pending reports, overdue routines, log gaps). Every agent gets this.

**scope-guard-files** (PreToolUse, Edit|Write) — Blocks file edits outside the agent's workspace. If the backend agent tries to edit a frontend file, the edit is rejected. Also blocks `.env` modifications. This is what makes multi-agent workspaces safe.

**scope-guard-commands** (PreToolUse, Bash) — For non-executor agents like tech-lead and security. Blocks running application commands (tests, builds, servers) to keep them in their advisory role.

**dispatch-guard** (PreToolUse, Agent) — Tech-lead only. Validates that agent dispatches include worktree isolation and a plan reference. Prevents ad-hoc dispatches without a plan.

**subagent-stop-review** (SubagentStop) — Tech-lead only. When a dispatched agent finishes, outputs a structured review checklist so the tech-lead evaluates the work before approving.

**api-security-check** (PreToolUse, Edit|Write) — Scans code changes for security anti-patterns: SQL injection, hardcoded secrets, `eval()`, CORS wildcards, disabled CSRF.

**test-integrity-guard** (PreToolUse, Edit|Write) — Catches test quality issues: removed assertions, `.skip()` / `.only()`, empty test bodies, debug artifacts like `console.log`.

**iac-safety-guard** (PreToolUse, Bash) — DevOps only. Blocks dangerous infrastructure commands: `terraform destroy`, `kubectl delete namespace`, unsafe Docker operations.

## Understanding Routines

Routines are periodic maintenance tasks — things that should happen on a schedule to keep the project healthy.

### How They Work

1. Each routine has a **frequency** (e.g., "every 7 days") and a **procedure** (concrete steps)
2. The routine dashboard (`agent/Core/routines.md`) tracks when each routine last ran and when it's next due
3. The **routine-check sensor** fires at session start, reads the dashboard, and flags overdue routines
4. You tell the agent to run the routine; it follows the procedure and updates the dashboard

The routine-check sensor is auto-managed — Bonsai adds it when you install routines and removes it when you remove all routines.

### Available Routines

| Routine | Frequency | What it does |
|---------|-----------|-------------|
| **dependency-audit** | 7 days | Scans dependencies for known CVEs and unmaintained packages |
| **vulnerability-scan** | 7 days | SAST scan, secrets scan, dependency cross-reference |
| **doc-freshness-check** | 7 days | Finds documentation that has drifted from recent code changes |
| **infra-drift-check** | 7 days | Compares declared IaC state against actual cloud resources |
| **status-hygiene** | 5 days | Archives old done items, validates pending items in Status.md |
| **memory-consolidation** | 5 days | Cleans up and validates working memory entries |
| **roadmap-accuracy** | 14 days | Ensures Roadmap.md reflects what's actually built and planned |

Not every agent gets every routine. Dependency-audit is for devops and security agents. Vulnerability-scan is for the security agent. The project-health routines (status, docs, memory, roadmap) are available to all agents.

## Choosing the Right Setup

### Solo Developer

**One fullstack agent** covers most needs. It handles UI, API, database, auth, and testing. Add the security-audit workflow if you want periodic security reviews.

```bash
bonsai init
bonsai add  # select fullstack
```

### Solo Developer with Planning Discipline

**Tech-lead + fullstack.** The tech-lead forces you to plan before building. You talk to the tech-lead to plan, then switch to the fullstack agent (or let the tech-lead dispatch) to execute.

```bash
bonsai init
bonsai add  # select tech-lead
bonsai add  # select fullstack
```

### Small Team / Complex Project

**Tech-lead + backend + frontend.** The tech-lead plans and reviews, specialized agents execute in their domain. Good when backend and frontend have meaningfully different conventions.

```bash
bonsai init
bonsai add  # select tech-lead
bonsai add  # select backend
bonsai add  # select frontend
```

### With Infrastructure

**Add devops** to any setup when you have Terraform, Kubernetes, CI/CD pipelines, or container configs. The devops agent comes with infrastructure-specific sensors (iac-safety-guard) and routines (infra-drift-check).

### Security-Conscious Projects

**Add security** to any setup. It comes with vulnerability-scan and dependency-audit routines, the api-security-check sensor, and the security-audit workflow. Good for regulated environments or when handling sensitive data.

### The Full Team

```bash
bonsai init
bonsai add  # tech-lead — plans, reviews, orchestrates
bonsai add  # backend — API, database, server logic
bonsai add  # frontend — UI, state management, styling
bonsai add  # devops — infrastructure, CI/CD, containers
bonsai add  # security — audits, scanning, compliance
```

## Tips and Best Practices

**Always start with "Hi, get started."** This one habit dramatically improves agent behavior. It ensures the agent processes its full startup context before working.

**Keep FieldNotes.md updated.** If you made changes outside a session, had a meeting, got new requirements, or just have context the agent should know — write it in FieldNotes. Agents read this every session.

**Let the tech-lead plan before you execute.** Even if you're tempted to tell the backend agent to "just add a login page," a 5-minute planning session catches edge cases, surfaces questions, and produces a clear implementation spec.

**Review plans before dispatching.** Plans are markdown files. Read them. Push back on approaches you disagree with. It's much cheaper to fix a plan than to fix implemented code.

**Reference plans by number.** "Execute plan 003" is unambiguous. "Do the auth work" might confuse the agent about which plan or scope you mean.

**Use the decision log.** When you settle an architectural question (PostgreSQL over MongoDB, REST over GraphQL, monorepo over polyrepo), have the agent log it. Future sessions will check this before proposing alternatives.

**Don't skip session wrap-up.** A quick "let's wrap up" at the end saves you from losing context. The agent updates its memory, writes a session log, and files any reports.

**Run overdue routines when flagged.** If the agent tells you dependency-audit is overdue, run it. These routines catch real issues — stale docs, vulnerable dependencies, drifted infrastructure.

**Use Reports/Pending/ as a review queue.** When code agents finish work, their reports go to Pending/. Make it a habit to review these, either yourself or through the tech-lead.

**Don't fight the scope boundaries.** If a sensor blocks an action, it's usually right. The backend agent shouldn't be editing frontend files. If you genuinely need cross-workspace changes, plan it through the tech-lead.

**Start small and grow.** You don't need every agent and every component from day one. Start with one or two agents. Add more as the project complexity demands it. `bonsai add` is designed to be run multiple times.
